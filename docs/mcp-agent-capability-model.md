# Capability Model

## Назначение

Документ фиксирует целевую модель возможностей агента для сценария автоподключения к MCP-серверам. Пользователь этой модели не должен знать о серверах, endpoint'ах и размещении конкретных `tools`.

## Основные принципы

1. Агент сам подключается к известным MCP-серверам при старте приложения.
2. Агент сам получает `serverInfo` и `tools/list` у каждого сервера.
3. Агент сам определяет, какие пользовательские команды доступны на основе найденных `tools`.
4. Пользователь работает только с прикладными командами CLI.
5. Выбор сервера и MCP-инструмента для выполнения команды делает агент.

## Минимальный стартовый набор серверов

Для первого прохода capability-модели агент знает один стартовый сервер:

- `local_mcp_server`
- endpoint: `http://127.0.0.1:3000/mcp`

Структура должна оставаться расширяемой для будущего добавления нескольких MCP-серверов без перестройки пользовательского контракта.

## Данные, которые агент хранит по серверу

После успешной подготовки агент должен хранить по каждому серверу:

- внутренний идентификатор сервера;
- endpoint;
- server info:
  - имя;
  - версия;
  - заголовок;
  - инструкции;
- список доступных MCP `tools`;
- признак успешности подключения;
- текст ошибки подготовки, если сервер недоступен.

## Реестр подключённых серверов

Агентный реестр должен быть рассчитан на несколько серверов, даже если на первом этапе сервер только один.

Минимально он должен позволять:

- получить все известные подключения;
- получить все успешно подготовленные подключения;
- найти сервер, на котором доступен конкретный MCP `tool`;
- исключать из пользовательских команд инструменты недоступных серверов.

## Индекс пользовательских команд

Агент хранит не только сырой список `tools`, но и производный индекс доступных пользовательских команд.

Минимальный состав записи о пользовательской команде:

- стабильный идентификатор команды;
- CLI-форма команды;
- пользовательское описание;
- MCP `tool`, который нужен для выполнения;
- сервер, через который этот `tool` должен вызываться.

## Правило публикации пользовательских команд

Пользовательская команда попадает в доступный список только если одновременно выполнены условия:

1. соответствующий MCP-сервер успешно подготовлен;
2. нужный MCP `tool` реально найден в `tools/list`;
3. агент знает, как преобразовать пользовательскую команду в аргументы вызова этого `tool`.

Если хотя бы одно условие не выполнено, команда не должна показываться пользователю как доступная.

## Стартовый набор поддерживаемых пользовательских команд

Для первого прохода capability-модели поддерживаются такие пользовательские сценарии:

- `tool posts`
- `tool post <postId>`

Они строятся из найденных MCP-инструментов так:

| Пользовательская команда | MCP tool | Назначение |
| --- | --- | --- |
| `tool posts` | `list_posts` | Показать первые 10 публикаций из JSONPlaceholder |
| `tool post <postId>` | `fetch_post` | Показать публикацию по идентификатору |

## Правило маршрутизации команды

Когда пользователь вызывает прикладную команду, агент выполняет такой порядок действий:

1. Находит запись команды в capability-индексе.
2. Берёт из неё связанный MCP `tool`.
3. Берёт из неё сервер, через который этот `tool` доступен.
4. Формирует аргументы вызова `tools/call`.
5. Делегирует инфраструктурный вызов в `mcp`-слой.

Пользовательский ввод не должен содержать:

- endpoint;
- идентификатор сервера;
- имя MCP-инструмента как обязательную инфраструктурную деталь.

## Правило выбора сервера

На первом проходе, когда нужный MCP `tool` найден ровно на одном сервере, агент использует этот сервер без дополнительной логики.

Если в будущем одинаковый `tool` окажется доступен на нескольких серверах, правило выбора должно оставаться внутри агента и не перекладываться на пользователя.

## Ошибки capability-уровня

Если capability-модель не может обслужить пользовательскую команду, верхним слоям должна возвращаться прикладная ошибка одного из типов:

- команда недоступна, потому что агент не завершил подготовку;
- команда недоступна, потому что соответствующий MCP `tool` не найден ни на одном готовом сервере;
- команда недоступна, потому что сервер есть, но нужный `tool` для неё не опубликован.

Эти ошибки должны объяснять пользователю проблему прикладным языком и не заставлять его вручную разбираться с endpoint'ами.
