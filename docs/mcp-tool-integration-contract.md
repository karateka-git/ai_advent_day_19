# Контракт первого MCP-инструмента

Документ фиксирует первый прикладной vertical slice для ТЗ `mcp-tool-integration-spec.md`.

## Выбранный Сценарий

Первым прикладным сценарием выбран вызов mock HTTP API через MCP-инструмент, который получает публикацию по её идентификатору и возвращает результат в читаемом виде.

Почему выбран именно этот вариант:

- он минимален по инфраструктуре;
- даёт реальный `tools/call`, а не только демонстрацию `tools/list`;
- легко воспроизводится локально;
- не требует авторизации и сложного окружения;
- остаётся расширяемым для следующих инструментов.

## Источник Данных

В качестве первого внешнего источника данных выбран публичный mock API `JSONPlaceholder`.

Первый endpoint:

- `GET https://jsonplaceholder.typicode.com/posts/{postId}`

Роль этого источника на текущем этапе:

- подтвердить саму интеграцию MCP tool с внешним HTTP API;
- дать простой, стабильный и понятный контракт ответа;
- оставить возможность позже заменить источник данных на другой HTTP API без изменения общей цепочки `presentation -> workflow -> agent -> mcp`.

## Первый Инструмент

Имя инструмента:

- `fetch_post`

Назначение:

- получить публикацию из mock API по идентификатору и вернуть её в удобной для CLI форме.

Черновое описание инструмента для MCP server:

- `Получает публикацию из mock API JSONPlaceholder по идентификатору.`

## Входной Контракт

Минимальный обязательный параметр:

- `postId: number`

Семантика:

- идентификатор публикации, которую нужно запросить во внешнем API.

Ожидаемая `inputSchema` на сервере:

```json
{
  "type": "object",
  "properties": {
    "postId": {
      "type": "number",
      "description": "Идентификатор публикации в JSONPlaceholder."
    }
  },
  "required": ["postId"]
}
```

## Ожидаемый Результат

Минимально полезный результат вызова должен содержать:

- идентификатор публикации;
- идентификатор автора;
- заголовок публикации;
- текст публикации.

На уровне проектных моделей результат удобно трактовать так:

```text
postId
userId
title
body
```

На уровне первого серверного ответа через `CallToolResult` достаточно вернуть читаемый текстовый блок с этими полями.

Пример ожидаемого читаемого результата:

```text
Публикация #1
Автор: 1
Заголовок: sunt aut facere repellat provident occaecati excepturi optio reprehenderit
Текст: quia et suscipit...
```

## Пользовательская Команда Приложения

Для первого прохода зафиксирована команда:

- `tool post <postId>`

Пример:

- `tool post 1`

Трактовка по слоям:

- `presentation` разбирает `tool post 1`;
- `workflow` создаёт команду вызова инструмента;
- `agent` инициирует `tools/call`;
- `mcp` вызывает MCP server;
- MCP server запрашивает публикацию во внешнем API и возвращает результат вверх по цепочке.

## Граница Текущего Этапа

На текущем этапе документ фиксирует только контракт и пользовательский сценарий.

Пока не входят в объём этого шага:

- код server-side HTTP-клиента;
- реализация `tools/call` в `McpClient`;
- новые `workflow` и `agent` модели;
- parser/formatter для команды `tool post <postId>`.
