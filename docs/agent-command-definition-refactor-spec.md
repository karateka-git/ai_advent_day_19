# Техническое задание

## Название

Рефакторинг capability-mapping агента через отдельные command definitions.

## Цель

Убрать из agent bootstrap текущую хрупкую схему, в которой:

- поддерживаемые пользовательские команды описаны анонимным списком внутри `buildAvailableAgentCommands(...)`;
- `commandId` и `serverId` представлены строками;
- выбор сервера для команды неявно зависит от `firstOrNull(...)` по списку подготовленных серверов.

Вместо этого нужно перейти к модели, в которой:

1. каждая поддерживаемая пользовательская команда описана отдельной command definition;
2. command definition сама знает:
   - свой стабильный идентификатор;
   - CLI-паттерн;
   - описание;
   - какой MCP `tool` ей нужен;
   - по какому правилу выбирать сервер;
3. агент получает список доступных пользовательских команд через отдельный resolver поверх command definitions;
4. `commandId` и `serverId` перестают быть "магическими строками" и становятся типизированными идентификаторами.

## Проблема

Текущее решение в `AgentCommandCatalog.kt` имеет несколько архитектурных слабостей:

- знание о пользовательских командах спрятано в top-level функции;
- декларация команды, правило маршрутизации и поиск сервера смешаны в одном месте;
- `servers.firstOrNull { ... }` делает порядок списка серверов скрытым источником поведения;
- перенос команды на другой сервер требует неявного изменения runtime-окружения, а не явного изменения самой команды;
- строковые `commandId` и `serverId` увеличивают риск опечаток и ухудшают рефакторинг.

## Целевая модель

После реализации рефакторинга проект должен работать так:

1. В agent bootstrap существует набор command definitions.
2. Каждая command definition описывает ровно одну пользовательскую команду.
3. Каждая command definition умеет по списку `PreparedMcpServer` определить:
   - доступна ли команда;
   - какой сервер должен быть выбран;
   - какой MCP `tool` будет вызван.
4. Отдельный resolver собирает список `AvailableAgentCommand` из command definitions.
5. Агент больше не строит доступные команды через глобальный builder по списку строковых описаний.
6. Для изменения целевого сервера конкретной команды достаточно изменить routing-правило в её definition.

## Архитектурные требования

### `agent/bootstrap/commands`

Должен содержать:

- базовый контракт command definition;
- отдельные реализации для конкретных пользовательских команд;
- resolver, который проходит по definitions и собирает доступные команды.

Не должен:

- хранить "магический" top-level список строковых команд без явного типа и ответственности;
- выбирать сервер через неявный `firstOrNull(...)` по всему списку серверов без участия самой команды.

### `agent/bootstrap/models`

Должен содержать:

- runtime-модели подготовленных серверов;
- runtime-модель доступной пользовательской команды;
- типизированные идентификаторы команд и серверов.

### `agent`

Должен:

- использовать resolver command definitions при `Prepare`;
- хранить в registry уже разрешённые команды;
- вызывать `CallAvailableCommand` по типизированному command id.

### `workflow`

Должен:

- использовать типизированные command id при обращении к агенту;
- не знать о server routing конкретной команды.

## Ограничения

- `README` на этом проходе не обновляется;
- пользовательский CLI-контракт пока остаётся прежним:
  - `tool posts`
  - `tool post <postId>`
- существующее поведение startup UX этапов capability bootstrap должно сохраниться;
- low-level `DefaultMcpClient` не рефакторится в этом ТЗ.

## Этапы реализации

### Этап 1. Формализация command-definition модели

Цель:

- зафиксировать новую модель refactor-а и структуру файлов;
- описать обязанности definition, resolver и типизированных идентификаторов.

Результат этапа:

- в проекте есть отдельное ТЗ по рефакторингу;
- создан unified implementation log;
- зафиксирована целевая схема command-definition слоя.

### Этап 2. Внедрение command-definition слоя

Цель:

- заменить текущий `buildAvailableAgentCommands(...)` на explicit command definitions и resolver.

Результат этапа:

- top-level builder удалён;
- поддерживаемые команды описываются отдельными definition-классами;
- availability строится через resolver, а не через единый анонимный список.

### Этап 3. Типизация `commandId` и `serverId` и явный routing

Цель:

- убрать строковые `commandId` и `serverId`;
- закрепить у command definition явное routing-правило.

Результат этапа:

- `commandId` и `serverId` типизированы;
- каждая command definition явно задаёт правило выбора сервера;
- выбор сервера больше не зависит от порядка списка серверов.

### Этап 4. Интеграция, тесты и синхронизация журналов

Цель:

- синхронизировать agent/workflow/tests с новой моделью;
- подтвердить, что поведение осталось рабочим.

Результат этапа:

- agent и workflow используют новую модель до конца;
- unit-тесты зелёные;
- unified implementation log закрывает все этапы этого ТЗ.

## Критерии готовности

Реализация считается завершённой, если:

- `AgentCommandCatalog.kt` в текущем виде больше не существует;
- пользовательские команды описываются отдельными command definitions;
- для конкретной команды можно явно увидеть, какой routing она использует;
- `commandId` и `serverId` больше не используются как строки в agent/workflow capability-потоке;
- `.\gradlew.bat test` проходит успешно;
- результат всех этапов зафиксирован в отдельном implementation log.
