# Техническое задание

## Название

Добавление параллельного stateful Streamable HTTP контура с SSE и push-сценарием `start_random_posts`.

## Цель

Добавить в проект отдельный stateful MCP runtime-контур, в котором:

1. текущий stateless reference остаётся рабочим и не деградирует;
2. рядом появляется stateful MCP server/runtime для push-сценариев;
3. пользователь через общий CLI и агентный слой запускает MCP tool `start_random_posts`;
4. stateful server до завершения клиентской сессии раз в заданный интервал отправляет в клиента случайную публикацию из `JSONPlaceholder`.

## Проблема

Текущий проект уже имеет рабочий stateless reference-контур с request/response-сценариями:

- клиент подготавливает capability snapshot и вызывает MCP tools по запросу;
- сервер отвечает только на `tools/list` и `tools/call`;
- после завершения запроса никакой активной server-to-client доставки результата не происходит;
- текущий runtime не позволяет показать "живого" агента, который сам присылает обновления в уже подключённый клиент.

Полный перевод всего проекта на stateful transport на этом этапе нежелателен, потому что:

- он создаёт риск регрессии для уже готового stateless reference;
- он размывает учебную границу между базовым MCP lifecycle и advanced push-сценарием;
- он усложняет проект сильнее, чем требует первый vertical slice с server push.

Из-за этого нужен отдельный stateful runtime-контур рядом с текущим stateless-контуром, а не замена одного другим.

## Выбранный Сценарий

Первым push-сценарием выбран периодический показ случайной публикации из уже подключённого mock API `JSONPlaceholder`.

Пользовательский эффект:

- пользователь запускает режим через `start_random_posts`;
- сервер подтверждает старт подписки;
- затем сервер раз в `intervalMinutes` отправляет в клиента новую случайную публикацию;
- остановка выполняется автоматически при завершении клиентской сессии.

Почему выбран именно этот сценарий:

- он переиспользует уже существующий API-клиент и домен публикаций;
- он проще полноценного scheduler/job-management;
- он демонстрирует stateful transport, SSE и server push без лишней доменной сложности;
- он оставляет естественный путь к будущему планировщику и фоновым задачам.

## Граница Текущего Этапа

В этот проход входят:

- добавление параллельного stateful server/client runtime-контурa рядом с текущим stateless;
- подключение SSE для server-to-client notifications;
- отдельный локальный endpoint для stateful-контура;
- серверный lifecycle клиентской сессии;
- MCP tool `start_random_posts(intervalMinutes = 5)`;
- один активный periodic random-post ticker на одну клиентскую сессию;
- push уведомлений в клиента в JSON-формате;
- автоматическая остановка ticker при завершении клиентской сессии;
- сохранение общего CLI, workflow и agent слоя;
- выделение `common`-зон внутри `mcp/client` и `mcp/server` для общих сущностей stateless/stateful реализаций.

В этот проход не входят:

- полный перевод всего проекта на stateful transport;
- отдельный `stop_random_posts` tool;
- универсальный scheduler для произвольных задач;
- хранение подписок или истории в SQLite/JSON;
- множественные параллельные random-post подписки в одной сессии;
- произвольные типы push-событий;
- отдельная система маршрутизации уведомлений между несколькими внешними клиентами.

## Целевая Модель

После реализации проект должен работать так:

1. В проекте существуют два runtime-подхода:
   - stateless reference для базовых MCP-сценариев;
   - stateful push runtime для server-to-client уведомлений.
2. Общий CLI, workflow и agent слой остаются едиными для пользовательского сценария.
3. Внутри `mcp/client` и `mcp/server` появляются `common`, `stateless` и `stateful` зоны ответственности.
4. Stateful MCP client устанавливает подключение к stateful MCP server.
5. Клиент держит активный SSE-поток для server-to-client событий параллельно с обычным CLI-вводом.
6. Stateful MCP server хранит runtime-информацию об активных клиентских сессиях.
7. Пользователь вызывает MCP tool `start_random_posts`.
8. Сервер регистрирует или обновляет random-post подписку для текущей клиентской сессии.
9. Для этой сессии запускается фоновый coroutine-based ticker.
10. На каждом тике сервер выбирает случайный `postId` из допустимого диапазона `1..100`.
11. Сервер получает публикацию через уже существующий `JsonPlaceholderApiClient`.
12. Сервер отправляет клиенту push-уведомление через SSE.
13. При завершении клиентской сессии связанная подписка и фоновые coroutine-ресурсы очищаются автоматически.

## Архитектурные Требования

### Transport

Проект не должен заменять существующий stateless transport.

Вместо этого рядом должен появиться отдельный stateful runtime-контур.

Stateless сервер должен:

- оставаться рабочим для существующего reference-сценария;
- не требовать семантических изменений ради нового push-сценария.

Stateful сервер должен:

- поддерживать stateful MCP session lifecycle;
- публиковать server-to-client события через SSE;
- поддерживать новый push-сценарий `start_random_posts`.

Клиентский слой должен:

- сохранять общий пользовательский CLI-сценарий;
- уметь работать с разными MCP runtime-реализациями под капотом;
- для stateful-контура держать активную сессию на протяжении пользовательской CLI-сессии;
- уметь принимать push-уведомления параллельно с обычными результатами вызова команд.

### Разделение Ответственности Между Подходами

В проекте должно быть явное разделение между:

- общим пользовательским контуром;
- stateless transport/runtime реализацией;
- stateful transport/runtime реализацией.

При этом:

- `presentation/cli`, `workflow` и `agent` не должны дублироваться только ради нового transport-подхода;
- различия между подходами должны жить в `mcp/client/...` и `mcp/server/...`;
- общие сущности для client/server внутри этих областей должны выноситься в `common`.

### Правило Маршрутизации Команд

На текущем этапе достаточным и обязательным правилом маршрутизации считается следующее:

- каждая пользовательская команда или MCP tool-сценарий явно знает, к какому runtime-контуру она относится;
- выбор между `stateless` и `stateful` не должен зависеть от неявного порядка endpoint-ов, fallback-логики или "угадывания" по runtime-состоянию;
- обычные существующие tool-сценарии продолжают маршрутизироваться в stateless-контур;
- `start_random_posts` явно маршрутизируется в stateful-контур.

Следствие для реализации:

- общий CLI/agent/workflow слой остаётся единым;
- но command-definition или равнозначный routing-слой обязан явно фиксировать backend/runtime для каждой поддерживаемой команды.

### Правило Для `common`

`common` используется как зона общих сущностей для `stateless` и `stateful` реализаций.

Туда следует выносить только те элементы, которые действительно переиспользуются обоими подходами.

Примеры подходящих кандидатов:

- общие модели connection snapshot;
- общие модели tool call;
- общие notification payload contracts;
- общие session value objects;
- общий доступ к `JSONPlaceholder`;
- общие server-side tool handlers, если они используются в обоих режимах.

Не следует выносить в `common`:

- stateful-specific session registries;
- SSE listeners;
- stateful subscriptions и ticker-runtime;
- stateless-specific transport wiring.

### Session Lifecycle

В stateful-контуре должен появиться runtime-слой управления клиентскими сессиями.

Он должен:

- различать клиентские сессии;
- связывать random-post подписку с конкретной сессией;
- очищать runtime-состояние при завершении сессии;
- не требовать отдельного ручного stop-инструмента на текущем этапе.

Дополнительно для текущего этапа нужно зафиксировать правила нештатного завершения:

- разрыв SSE или потеря stateful-соединения должны рассматриваться как сигнал к завершению соответствующей session-bound подписки;
- сервер должен уметь освобождать orphan runtime-состояние и останавливать связанный ticker;
- reconnect не должен автоматически восстанавливать старую подписку, если это отдельно не реализовано;
- повторное подключение после разрыва трактуется как новая клиентская сессия.

### Random Post Subscription

На одну клиентскую сессию допускается не более одной активной random-post подписки.

Семантика `start_random_posts`:

- если подписки ещё нет, она создаётся;
- если подписка уже есть, её интервал обновляется;
- второй независимый ticker для той же сессии не создаётся.

### Push Notification Contract

Push-уведомления должны передаваться не как голый текст, а как простая JSON-модель.

На текущем этапе модель содержит одно поле:

```json
{
  "message": "..."
}
```

Требования к этой модели:

- поле `message` обязательно;
- значение должно быть человекочитаемым summary случайной публикации;
- модель должна быть расширяемой в будущем без смены общего push-подхода.

На protocol layer для текущего этапа нужно зафиксировать ещё и notification envelope:

- push-сообщение доставляется как server-to-client MCP notification;
- клиент должен отличать push-уведомление от обычного результата `tools/call` по notification method, а не по эвристикам в текстовом выводе;
- конкретное имя notification method фиксируется как `notifications/random_post`;
- это имя должно быть зафиксировано в реализации и тестах как часть stateful push-контракта.

### Random Post Retrieval

Для первого прохода источник случайности фиксируется явно:

- случайный `postId` выбирается из диапазона `1..100`;
- публикация запрашивается через существующий server-side клиент `JsonPlaceholderApiClient`.

Если публикация по выбранному идентификатору не получена, сервер может:

- повторить попытку на следующем тике;
- либо отправить диагностическое сообщение через тот же notification-поток.

Главное требование текущего этапа:

- фоновый ticker не должен падать целиком из-за единичной ошибки получения публикации.

### Целевая Структура Пакетов

На уровне итогового проектного устройства целевая структура должна отражать три зоны:

- `common`
- `stateless`
- `stateful`

Применительно к MCP-слою ожидается следующая схема:

```text
src/main/kotlin/ru/compadre/mcp/
  presentation/
    cli/

  workflow/
    ...

  agent/
    ...

  mcp/
    client/
      McpClient.kt
      common/
        model/
        toolcall/
          model/
        notifications/
        session/
      stateless/
      stateful/

    server/
      common/
        api/
          jsonplaceholder/
        toolcall/
          tools/
            fetchpost/
            listposts/
        notifications/
        session/
      stateless/
      stateful/
        session/
        subscriptions/
        background/
        toolcall/
          tools/
            startrandomposts/
```

Смысл этой структуры:

- верхний пользовательский слой остаётся общим;
- transport/runtime вариативность локализована внутри `mcp/client` и `mcp/server`;
- общий код для обоих подходов выносится в `common`;
- режим-специфичный код остаётся внутри `stateless` и `stateful`.

### Endpoint И Notification Method

Для первого прохода фиксируются следующие значения:

- stateless-контур продолжает использовать текущий локальный endpoint;
- stateful-контур публикуется на отдельном локальном endpoint `http://127.0.0.1:3001/mcp`;
- push-события random-post доставляются через notification method `notifications/random_post`.

Следствие для реализации:

- stateful runtime не конфликтует с текущим stateless server по порту;
- клиент может однозначно различать random-post push-события по имени notification method;
- это имя считается частью проектного stateful push-контракта.

## MCP Tool Contract

### Имя Инструмента

- `start_random_posts`

### Назначение

- включает или обновляет periodic random-post push для текущей клиентской сессии.

### Входной Контракт

Инструмент принимает один параметр:

- `intervalMinutes: integer`

Правила:

- значение по умолчанию: `5`;
- минимально допустимое значение: `1`;
- значение интерпретируется как целое число минут.

Черновой `inputSchema`:

```json
{
  "type": "object",
  "properties": {
    "intervalMinutes": {
      "type": "integer",
      "minimum": 1,
      "description": "Интервал в минутах между push-отправками случайных публикаций. Минимум 1, по умолчанию 5."
    }
  }
}
```

### Ожидаемый Результат Вызова

Немедленный результат `tools/call` должен только подтверждать, что режим активирован или обновлён.

Пример читаемого результата:

```text
Random posts push активирован для текущей сессии. Интервал: 5 мин.
```

Дальнейшие публикации должны приходить уже не как продолжение `tools/call`, а как отдельные server push notifications.

## Клиентское Поведение

CLI-клиент должен:

- сохранить текущий интерактивный сценарий;
- продолжать принимать пользовательские команды;
- параллельно слушать SSE-уведомления;
- печатать push-сообщения в консоль так, чтобы они были различимы от обычных результатов команд.

На текущем этапе не требуется:

- сложная UI-разметка;
- история push-сообщений;
- отдельный экран управления подписками.

## Этапы Реализации

### Этап 1. Спецификация parallel runtime-модели

Цель:

- зафиксировать целевую модель сосуществования `stateless` и `stateful` подходов, поведение клиентской сессии и контракт первого push-tool.

Результат этапа:

- в проекте есть отдельное ТЗ по parallel stateful runtime и random-post push;
- зафиксированы ограничения первого vertical slice;
- определён notification contract c JSON payload `message`;
- зафиксирована целевая структура `common/stateless/stateful`.

### Этап 2. Выделение runtime-контуров и common-структуры

Цель:

- разделить MCP client/server слой на `common`, `stateless` и `stateful` зоны ответственности без дублирования общего CLI/agent/workflow.

Результат этапа:

- общие сущности перенесены в `common`;
- stateless runtime остаётся рабочим;
- в проекте подготовлена точка расширения для stateful runtime;
- для пользовательских команд зафиксировано явное routing-правило по runtime-контурам.

### Этап 3. Stateful transport и session lifecycle

Цель:

- реализовать stateful Streamable HTTP контур, SSE и runtime-модель активных клиентских сессий.

Результат этапа:

- stateful server публикуется отдельно от stateless server;
- stateful server использует endpoint `http://127.0.0.1:3001/mcp`;
- stateful client умеет держать долгоживущую MCP-сессию;
- сервер умеет регистрировать и очищать stateful client sessions;
- на одну сессию поддерживается одна active random-post подписка;
- завершение сессии корректно останавливает связанные фоновые задачи;
- обрыв соединения и orphan session cleanup покрыты acceptance-сценарием.

### Этап 4. MCP tool `start_random_posts`

Цель:

- зарегистрировать новый MCP tool и реализовать его server-side поведение.

Результат этапа:

- `tools/list` показывает `start_random_posts`;
- `tools/call` активирует или обновляет random-post подписку текущей сессии;
- повторный старт не создаёт дубль ticker для той же сессии;
- повторный `start_random_posts` обновляет интервал текущей подписки, а не создаёт новую.

### Этап 5. Background ticker и push notifications

Цель:

- реализовать фоновую coroutine-задачу, которая периодически получает случайную публикацию и отправляет её в клиента.

Результат этапа:

- сервер раз в `intervalMinutes` выбирает случайный `postId`;
- публикация извлекается через `JsonPlaceholderApiClient`;
- клиент получает push-событие через SSE;
- push-событие передаётся как JSON-модель с полем `message`;
- notification method `notifications/random_post` и payload трактуются как часть тестируемого контракта.

### Этап 6. Интеграция клиента, тесты и документация

Цель:

- встроить push-сценарий в общий CLI-клиент, покрыть stateless/stateful runtime unit/integration-тестами и синхронизировать docs.

Результат этапа:

- CLI-клиент параллельно принимает команды и push-уведомления;
- тесты подтверждают регистрацию нового tool, работоспособность stateful transport и cleanup по завершению сессии;
- тесты покрывают:
  - успешную SSE-подписку;
  - повторный `start_random_posts`;
  - обрыв соединения;
  - остановку ticker при cleanup;
  - ошибку получения публикации из `JsonPlaceholder`;
- stateless reference-сценарии остаются зелёными;
- README и связанные документы отражают сосуществование stateless и stateful подходов и сценарий `start_random_posts`.

## Критерии Готовности

Реализация считается завершённой, если:

- в проекте сосуществуют stateless и stateful runtime-подходы;
- stateless reference остаётся рабочим;
- stateful server и client работают в stateful Streamable HTTP режиме;
- stateful server публикуется на `http://127.0.0.1:3001/mcp`;
- SSE push-сообщения доходят до интерактивного CLI-клиента;
- MCP tool `start_random_posts` доступен через `tools/list`;
- вызов `start_random_posts` активирует или обновляет random-post подписку текущей сессии;
- сервер периодически отправляет случайные публикации из `JSONPlaceholder` без новых пользовательских запросов;
- push-уведомления используют JSON payload c полем `message`;
- `intervalMinutes` валидируется как `integer >= 1`;
- повторный `start_random_posts` обновляет интервал существующей подписки для текущей сессии;
- notification method для push-событий зафиксирован как `notifications/random_post` и различим для клиента;
- завершение клиентской сессии останавливает random-post ticker;
- обрыв соединения не оставляет висячие ticker/coroutine-ресурсы;
- общий CLI/agent/workflow слой не дублируется без необходимости;
- структура `common/stateless/stateful` отражена в MCP-слое проекта;
- минимальный MCP happy-path и существующие tool-сценарии не деградируют;
- `.\gradlew.bat test` проходит успешно;
- документация проекта синхронизирована с новой stateful-моделью.
