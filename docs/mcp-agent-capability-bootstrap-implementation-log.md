## Этап 1. Формализация агентной модели возможностей

Статус: завершён

Цель этапа:

- зафиксировать новую модель ответственности без пользовательского `connect`;
- описать, какие capabilities агент собирает при старте;
- описать, как пользовательские команды выводятся из MCP `tools`, а не из знания пользователя о серверах.

Выполненные действия:

1. Для `Этапа 1` в `docs/mcp-agent-capability-bootstrap-spec.md` добавлены подэтапы внедрения, чтобы явно разделить:
   - уточнение архитектурной границы слоёв;
   - фиксацию capability-модели агента;
   - синхронизацию верхнеуровневой документации.
2. В отдельном документе `docs/mcp-agent-capability-model.md` зафиксирована целевая capability-модель агента.
3. Для первого прохода письменно закреплён минимальный стартовый набор:
   - один известный MCP-сервер `local_mcp_server`;
   - пользовательские команды `tool posts` и `tool post <postId>`.
4. Зафиксировано, что пользователь не должен знать:
   - на каком сервере расположен `tool`;
   - какой endpoint используется;
   - как агент выполняет discovery.
5. В README добавлены ссылки на новое ТЗ, capability-модель и implementation log, а также обновлён следующий целевой сценарий развития.

Принятые решения:

- считать пользовательский `connect` устаревающей частью UX и не закладывать его в новую capability-модель;
- строить новый контракт вокруг agent-level discovery и agent-level маршрутизации команд;
- сразу проектировать модель хранения возможностей как расширяемую для нескольких MCP-серверов, даже если в первом проходе сервер только один.

Проверка:

- в проекте есть отдельное ТЗ по новой модели: `docs/mcp-agent-capability-bootstrap-spec.md`;
- в проекте есть отдельный capability-документ: `docs/mcp-agent-capability-model.md`;
- README ссылается на оба документа и на единый implementation log этого ТЗ.

Коммиты этапа:

- `597baf3` — детализация этапа 1 автоподготовки агента;
- `5c6a80c` — фиксация capability-модели агента;
- текущий коммит этапа — создание implementation log и синхронизация README с новым ТЗ.

Следующий шаг:

## Этап 2. Стартовая подготовка агента

Статус: завершён

Цель этапа:

- внедрить в агент явный сценарий подготовки MCP-возможностей;
- подключить к этому сценарию список известных серверов;
- сохранить результат подготовки в agent-level состоянии для следующих этапов.

Выполненные действия:

1. Для `Этапа 2` в `docs/mcp-agent-capability-bootstrap-spec.md` добавлены подэтапы внедрения, чтобы явно зафиксировать:
   - новый agent-level сценарий подготовки;
   - модели известных серверов и результатов подготовки;
   - требование к частично успешному discovery.
2. В agent-слое добавлен новый запрос `AgentRequest.Prepare`.
3. В agent-слое добавлен новый успешный ответ `AgentResponse.PreparationSuccess`.
4. Введены agent-level модели подготовки:
   - `KnownMcpServer`;
   - `PreparedMcpServer`;
   - `AgentCapabilitySnapshot`.
5. Добавлен `AgentCapabilityRegistry`, который хранит последний snapshot подготовленных возможностей.
6. `DefaultAgent` теперь умеет:
   - проходить по известным MCP-серверам;
   - вызывать `mcpClient.connect(...)` для каждого из них;
   - сохранять успешные и неуспешные результаты в один snapshot;
   - обновлять agent-level registry после завершения подготовки.
7. В `McpProjectConfig` добавлен стартовый список известных MCP-серверов для capability bootstrap.
8. Workflow-слой и архитектурные тесты синхронизированы с новым вариантом `AgentResponse`, чтобы компиляция и контракт оставались целостными.
9. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- на этапе подготовки не падать из-за одного недоступного сервера, а сохранять его в snapshot как неуспешный;
- хранить результат последней подготовки в отдельном `AgentCapabilityRegistry`, а не размазывать состояние по `App` или `workflow`;
- пока не связывать capability bootstrap с пользовательским стартовым UX, а сначала завершить agent-level основу.

Проверка:

- `DefaultAgent` умеет обрабатывать `AgentRequest.Prepare`;
- полностью успешная подготовка возвращает `PreparationSuccess` и обновляет registry;
- частично неуспешная подготовка сохраняет доступные серверы и ошибки недоступных в одном snapshot;
- `.\gradlew.bat test` завершается успешно.

Коммиты этапа:

- `937ff6b` — детализация этапа 2 capability bootstrap;
- `06e5770` — добавление стартовой подготовки MCP-возможностей в агент;
- текущий коммит этапа — фиксация результата этапа 2 в implementation log.

Следующий шаг:

- перейти к `Этапу 3` и построить capability-реестр с маршрутизацией пользовательских команд на нужный MCP `tool` и сервер.

## Этап 3. Реестр возможностей и маршрутизация команд

Статус: завершён

Цель этапа:

- построить agent-level capability-реестр пользовательских команд;
- научить агента находить нужный MCP `tool` и сервер по прикладной команде;
- убрать знание о маршрутизации из пользовательской стороны модели.

Выполненные действия:

1. Для `Этапа 3` в `docs/mcp-agent-capability-bootstrap-spec.md` зафиксировано, что маршрутизация пользовательской команды должна происходить через capability registry агента.
2. В capability-модели агента введена новая сущность `AvailableAgentCommand`.
3. В `AgentCapabilitySnapshot` добавлен список доступных пользовательских команд `availableCommands`.
4. В `AgentCapabilityRegistry` добавлен поиск команды по её стабильному идентификатору.
5. Добавлен `AgentCommandCatalog`, который:
   - знает поддерживаемые пользователю сценарии;
   - сопоставляет их с MCP `tools`;
   - строит список доступных команд из подготовленных серверов.
6. В agent-контракты добавлен новый запрос `AgentRequest.CallAvailableCommand`.
7. `DefaultAgent` теперь умеет:
   - искать прикладную команду в capability-реестре;
   - находить связанный MCP `tool` и сервер;
   - вызывать `tools/call` через уже вычисленный маршрут;
   - возвращать прикладную ошибку, если команда недоступна или preparation ещё не выполнен.
8. В `DefaultAgentTest` добавлены unit-тесты на:
   - наполнение capability snapshot доступными командами;
   - маршрутизацию `tool.post` на нужный endpoint и `fetch_post`;
   - ошибку вызова команды до подготовки.
9. Архитектурные тесты синхронизированы с новыми request/response-контрактами.
10. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- использовать стабильные agent-level идентификаторы команд:
  - `tool.posts`
  - `tool.post`
- хранить в capability-реестре уже готовую привязку:
  - пользовательская команда;
  - MCP `tool`;
  - сервер;
  - endpoint;
- пока выбирать первый подготовленный сервер, на котором найден нужный `tool`, оставляя более сложную стратегию выбора на будущее.

Проверка:

- capability snapshot после подготовки содержит список доступных команд;
- агент умеет маршрутизировать вызов `tool.post` через capability-реестр;
- вызов прикладной команды до подготовки возвращает понятную agent-level ошибку;
- `.\gradlew.bat test` завершается успешно.

Коммиты этапа:

- `be5f151` — добавление capability-реестра и маршрутизации команд в агент;
- текущий коммит этапа — фиксация результата этапа 3 в implementation log.

## Этап 4. Перестройка workflow и presentation под новую модель

Статус: завершён

Цель этапа:

- убрать пользовательский `connect` из `workflow` и `presentation`;
- перевести верхние слои на agent-level вызов доступных команд по `commandId`;
- подготовить presentation-friendly результат стартовой подготовки агента для следующего этапа UX.

Выполненные действия:

1. Для `Этапа 4` в `docs/mcp-agent-capability-bootstrap-spec.md` добавлены подэтапы внедрения, чтобы явно зафиксировать:
   - удаление `ConnectCommand` и `ConnectResult` из пользовательского CLI-контракта;
   - перевод `workflow` на `AgentRequest.CallAvailableCommand`;
   - передачу результата подготовки агента в presentation без server-level деталей.
2. Из `workflow` удалены пользовательские контракты `ConnectCommand` и `ConnectResult`.
3. В `workflow` добавлен новый bridge-сценарий `PrepareAgentCommand -> AgentPreparationResult`, который переводит `AgentResponse.PreparationSuccess` в presentation-friendly список доступных CLI-команд.
4. `ToolPostCommand` и `ToolPostsCommand` больше не хранят endpoint и не знают о конкретных MCP `toolName`.
5. `DefaultWorkflowCommandHandler` теперь:
   - делегирует `tool post` и `tool posts` через `AgentRequest.CallAvailableCommand`;
   - возвращает `ToolCallResult` в терминах пользовательской команды, а не в терминах endpoint/tool.
6. `ToolCallResult` упрощён до пользовательского контракта:
   - `commandText`;
   - `successful`;
   - `content`;
   - `errorMessage`.
7. `DefaultCliCommandParser` больше не принимает и не распознаёт `connect`, а разбирает только прикладные команды:
   - `tool posts`;
   - `tool post <postId>`.
8. `DefaultCliOutputFormatter` больше не форматирует `ConnectResult` и не показывает server-level детали маршрутизации. Вместо этого он:
   - умеет выводить `AgentPreparationResult`;
   - форматирует ошибки и успех `ToolCallResult` через пользовательскую команду.
9. Из `App` убраны:
   - parser dependency на default endpoint;
   - presentation-level флаг `isConnected`;
   - guard на ручной `connect` перед `tool ...`;
   - help-описание команды `connect`.
10. Обновлены unit-тесты `workflow` и `presentation` под новый CLI-контракт.
11. Выполнен прогон `.\gradlew.bat test`.

Принятые решения:

- на этом этапе окончательно считать `connect` внутренней инфраструктурной операцией, а не пользовательским сценарием;
- не выводить в CLI endpoint и MCP `toolName`, даже если они присутствуют в agent/mcp-слоях;
- подготовить в `workflow` и `presentation` отдельный результат `AgentPreparationResult`, чтобы следующий этап UX уже опирался на новый контракт, а не лез напрямую в agent-state.

Проверка:

- parser отклоняет `connect` и принимает только `tool posts` и `tool post <postId>`;
- `workflow` вызывает агент через `CallAvailableCommand`, а не через прямой `CallTool` с endpoint/toolName;
- formatter не печатает server-level данные при вызове прикладной команды;
- `.\gradlew.bat test` завершается успешно.

Коммиты этапа:

- `a5620b1` — детализация этапа 4 capability bootstrap;
- `2b4bf26` — удаление пользовательского `connect` из workflow/presentation и перевод CLI на available-command модель;
- текущий коммит этапа — фиксация результата этапа 4 в implementation log.

Следующий шаг:

## Этап 5. UX подготовки агента

Статус: завершён

Цель этапа:

- перенести подготовку агента в стартовый UX приложения;
- показать пользователю явное состояние загрузки до появления prompt;
- завершать запуск финальным приветствием только после завершения discovery.

Выполненные действия:

1. Для `Этапа 5` в `docs/mcp-agent-capability-bootstrap-spec.md` добавлены подэтапы внедрения, чтобы явно закрепить:
   - запуск `PrepareAgentCommand` до интерактивного цикла;
   - анимацию строки `Подготовка агента...`;
   - единое поведение для interactive и one-shot запуска;
   - завершение запуска с ошибкой при неуспешной подготовке.
2. В [App.kt](/C:/Users/compadre/Downloads/Projects/AiAdvent/day_17/src/main/kotlin/ru/compadre/mcp/App.kt) добавлен явный lifecycle-шаг `prepareAgent(...)`, который:
   - запускает `PrepareAgentCommand` до prompt;
   - крутит анимацию точек в одной строке;
   - очищает строку загрузки после завершения подготовки;
   - возвращает `AgentPreparationResult` для дальнейшего UX.
3. Стартовый сценарий приложения перестроен так, что:
   - при успешной подготовке в interactive-режиме сначала печатается итоговое приветствие агента и найденные команды;
   - затем показывается подсказка `Введите help...` и только после этого стартует интерактивный цикл;
   - при неуспешной подготовке приложение печатает прикладную ошибку и завершает запуск без prompt.
4. `help` в интерактивном режиме теперь строится из `availableCommands`, полученных после подготовки агента, а не из жёстко зашитого списка.
5. Для единообразного project workflow обновлены скрипты:
   - [scripts/invoke-client-commands.ps1](/C:/Users/compadre/Downloads/Projects/AiAdvent/day_17/scripts/invoke-client-commands.ps1)
   - [scripts/start-manual-check.ps1](/C:/Users/compadre/Downloads/Projects/AiAdvent/day_17/scripts/start-manual-check.ps1)
   чтобы headless/manual-check больше не слали устаревший `connect`.
6. Выполнены:
   - `.\gradlew.bat test`
   - `powershell -ExecutionPolicy Bypass -File .\scripts\start-manual-check.ps1 -Headless`

Принятые решения:

- анимацию подготовки держать минимальной и однострочной через `\r`, чтобы не засорять консоль;
- не печатать итоговое приветствие в one-shot режиме, но всё равно выполнять ту же подготовку до выполнения команды;
- считать failure подготовки блокирующей ошибкой запуска, потому что без capability snapshot пользовательский контракт ещё не сформирован.

Проверка:

- `.\gradlew.bat test` завершается успешно;
- headless manual-check показывает:
  - строку `Подготовка агента...` с анимацией;
  - итоговое приветствие `Агент готов к работе.`;
  - список доступных команд;
  - корректный `help` без `connect`.

Коммиты этапа:
